# Benchmark results

Generated by `benchmark/score.py` against the golden corpus (86 cases,
spec_version 0.2.0). Run date: 2026-07-08. spaCy models: en_core_web_lg/sm 3.8.0.

| scanner | recall (caught/total) | over-redaction (FP/guards) | notes |
|---|---|---|---|
| naive-regex | 7/46 (15%) | 0/21 (0%) | email/phone/ssn/ip + 3 key prefixes |
| detect-secrets | 36/37 (97%) | 21/21 (100%) | entropy/regex plugins; line-level |
| presidio-vanilla | 24/27 (88%) | 8/21 (38%) | default recognizers, no allow-list/thresholds |
| gitleaks | 7/37 (18%) | 0/21 (0%) | CLI, --no-git |
| trufflehog | 0/37 (0%) | 0/21 (0%) | CLI, --no-verification (verification-oriented; synthetic corpus) |
| cp-scrub (reference) | 65/71 (91%) | 1/21 (4%) | authored the corpus → ≈100% by construction |

## Reading

The corpus exposes a **precision/recall frontier that no off-the-shelf tool sits well
on for conversation scrubbing**:

- **detect-secrets** has the recall (97%, entropy-driven) but flags **every** guard —
  git SHAs, UUIDs, SHA-256 digests, base64 blobs, and all the code identifiers. A
  scrubber that redacts 100% of benign identifiers would shred a coding conversation;
  this is the over-redaction-drives-abandonment failure mode in one number.
- **gitleaks** has perfect precision here (0 false positives) but 18% recall — it
  allow-lists known *example* keys (e.g. `AKIA…EXAMPLE`) and its rules miss the
  synthetic vendor/structural families. It is a *repo* scanner, not a conversation one.
- **trufflehog** fires on nothing: its detectors target real, verifiable credentials,
  and the corpus is synthetic-only by hygiene. Out of domain → a floor, not a verdict.
- **Presidio (vanilla)** does well on PII (88%) but over-redacts a third of the guards
  (NER reads code identifiers as entities).
- **naive regex** is the "afternoon scrub": misses 85%, but is too simple to false-
  positive.

## What it caught in *our own* scrubber

The reference is not exempt. The June run surfaced 3 over-redactions in the full
`PresidioScrubber` — **Presidio's URL recognizer flagging dotted Java identifiers** as
schemeless URLs (`import com.google.common.collect.ImmutableList` →
`import [URL]mutableList`). Fixed since (URL removed from the Presidio entity map;
regex owns URL reduction), dropping the reference's corpus over-redaction from 26%
(5/19) to **4% (1/21)** — the benchmark surfaced a real precision bug, the corpus
working as designed.

## Over-redaction at scale (real code)

`benchmark/score_thestack.py` runs the scrubbers over real code files from
`bigcode/the-stack-smol-xs` (ungated). Real source is mostly not secret, so a high
flag-rate ≈ over-redaction.

**22 languages, 2200 files (2026-07-08):**

| tool | files flagged / 2200 | rate |
|---|---|---|
| naive-regex | 185 | 8% |
| **detect-secrets** | **2191** | **99%** |
| cp-scrub (all redactions) | 836 | 38% |
| cp-scrub (`[SECRET]` only) | **202** | **9%** |

detect-secrets flags **99%** of real code files — every hash, UUID, and long token; it
would redact almost every file. cp-scrub's *total* 38% includes legitimate URL/path/
email reduction; its **`[SECRET]`-only rate is ~9%**, ~11× lower. Critically, that 9%
is **essentially all bare base64/high-entropy blobs, not code identifiers** — of the
flagged tokens, 0 are pure-hex, 0 are camelCase identifiers, and 96% sit in no
integrity/key/base64 context. cp-scrub is not shredding variable names; it redacts
opaque ≥28-char base64 blobs that are shape-identical to a bare secret.

### Correction: the earlier "2%" was stale

The June run of this probe reported an `[SECRET]`-FP of **2%** (800 files, 8 langs). A
`git bisect` on the current tree found that regressed to **8.6% on the identical 8
languages and same cached files** — a code change, not a data or language effect.
Culprit: `071510d` ("gate benign-token rejector on a real dictionary, not a vowel
heuristic"), which correctly closed a leak (bare AWS-secret-key-shaped tokens were
excused because their incidental letter runs are vowel-bearing) at a precision cost
that was never re-benchmarked. The scrubber got *safer*; this doc went stale.

### The precision/recall frontier (entropy-pass min-length)

Raising the base64/dotted minimum length lowers the-stack over-redaction (28→32:
8.6%→8.0%), but it is **not free**. An initial sweep suggested "zero recall cost,"
but that measured recall only on the *Python client* scrub surface — which has a
generic `{24,}`-token backstop the entropy passes fall through to. Two things break
above 28:

- **Client**: a base64 secret carrying `+//` that the generic pass can't catch
  (`secret-entropy-base64-plusslash`, ~30 chars) leaks.
- **Worker gate** — the real trust boundary, which deliberately has **no**
  generic-entropy pass — leaks *every* 28–31-char entropy secret the moment the
  threshold rises (`secret-entropy-with-digit`, 31 chars). The gate is fully
  dependent on this threshold.

Bare benign base64 and bare unknown secrets overlap at 28–31 chars, and the gate has
no backstop there — so **there is no safe threshold tune.** An attempted 28→32 bump
was caught by the client + worker corpus suites (both recall tests failed) and
reverted. Recovering precision needs a real discriminator (decoded-entropy / n-gram /
SRI-lockfile exemptions) — a post-launch research item, not a length knob.

## Independent recall (real code + valid formats)

Corpus recall above is self-authored. Three independent checks triangulate it:

**Seeded recall on real code** — the cleanest test: inject known, *valid-format* PII
(Luhn-valid cards, real vendor key prefixes, US + international phones, proper emails)
into real `the-stack-dedup` files and measure catch-rate. Real context, no synthetic
text, formats controlled to be valid — so it isolates genuine recall from data artifacts.

| injected PII (valid format) | recall on real code |
|---|---|
| email · AWS/OpenAI/GitHub keys · IPv4 | **100%** |
| US phone · SSN | **100%** |
| **credit card (Luhn-valid)** | **100%** |
| **international phone** (`+44 20 7946 0958`) | **100%** — after the E.164 pattern (was 0%) |

Every real-format PII/secret type is caught. The one gap it surfaced — **international
phone at 0%** (US-centric NANP regex) — is now closed by a `+`-anchored E.164 pattern
(0.2% FP on real code, ReDoS-safe, corpus-pinned).

**Kaggle PII** (Learning Agency Lab, CC-BY, 600 human-labeled student essays) — real
prose, human-labeled, no synthetic artifacts. On cp-scrub's in-scope PII: **email 100%
(29/29), personal URL 97% (48/49), names 80% (687/853, NER on prose)** — **82% in-scope
aggregate**. Out-of-scope categories are correctly preserved (generic `ID_NUM`,
`USERNAME`). Corroborates the ~80–85% NER name-recall from a second, human-labeled source.

**ai4privacy** (`pii-masking-200k`, Apache-2.0, 1,200 samples) as a breadth cross-check:
73% in-scope PII, **100% email/IP, 86% names**. Its lower phone/CC figures were **synthetic
artifacts**, not real gaps — Faker generates non-Luhn "credit cards" (cp-scrub correctly
ignores them; the Luhn-valid seeded test is 100%) and exotic international phone formats.
NER labels scored with `en_core_web_lg`; approximate span-in-output leak check.

## Known limitations of the benign-token rejector

The dictionary-based rejector (`_looks_benign`, since `071510d`) is a deliberate
precision/recall trade, gitleaks/detect-secrets style:

- **Precision:** bare, contextless base64 blobs in real code (encoded data, inline
  assets) are redacted as `[SECRET]` — indistinguishable from a bare credential without
  decoding or context. This is the bulk of the ~9% real-code over-redaction above.
- **Recall:** a *bare* word-structured secret value with no keyword context
  (`PasswordResetTokenAbc123`) is still excused — the research-acknowledged trade of any
  word-based FP reduction. Mitigated by the keyword-gated `AUTH_TOKEN=` pass (not
  rejector-gated) plus the vendor/prefixed families, which are unaffected.
- Closing either further needs a decoded-entropy / n-gram pass — a later refinement.
